git rebase <branch-1> <branch-2> - перебазировать <branch-2> на вершину ветки <branch-1> (данная команда имеет состояние: в процессе или завершено)
git rebase <branch> - перебазировать текущую ветку на вершину ветки <branch> (данная команда имеет состояние: в процессе или завершено)
--флаги git rebase ...
--abort - прекратить rebase, сбросить состояние, и перенести HEAD на прежнее место (git reset --hard не переносит HEAD)
--quit - прекратить rebase, сбросить состояние, не возвращает HEAD (аналог: git reset --hard)
--skip - пропустить текущий коммит
--continue - продолжить rebase
--onto <branch> <commit> - перебазировать текущую ветку, начиная с <commit> (а не с начала ее ответвления от <branch> [сам <commit> не будет включен в rebase, будут включены коммиты выше него по истории]) на ветку <branch>. --onto нужно указывать перед веткой, на которую перебазируем
-x '<tests> - запускает тесты перед перебазированием каждого коммита, если успешно - перебазировать, иначе - правим коммиты (git commit --amend) и используем флаги выше. Так для каждого коммита
--preserve-merges (-p) - включить коммиты сляиния в перебазирование (существует проблема перебазирования коммитов слияния*)
-i - интерактивный rebase**, можно перебазировать на саму себя, чтобы просто переписать историю коммитов***
--autosquash - в интерактиве добавить коммиты-заплатки в сразу после целевых коммитов


P.S.: rebase по сути похож на merge, но имеет ряд отличий
Плюсы: упрощает работу с историей (потому что нет коммитов слияния)
Минусы: для безопасности командной разработки (удаленный ветки), лучше использовать только для приватных веток. Также возможны ошибки в коммитах
